---
title: 'Verify a complaint'
description: 'Police, bank, and court officers use this endpoint to record their verification decision on a complaint.'
---

<Note>
  This endpoint requires authentication. Pass the JWT token returned by [Log in](/api/auth/login) in the `token` request header. The caller's role is determined from the token and controls which verification field is updated.
</Note>

## Request

**Method:** `POST`  
**Path:** `/formupload/verifybypolice`

<Tip>
  Despite the endpoint name, this single route handles verification for all three verifying roles — Police, Bank, and Court. The server reads the caller's `usertype` from the JWT token and updates the appropriate field automatically.
</Tip>

### Header parameters

<ParamField header="token" type="string" required>
  A valid JWT session token obtained from the [login endpoint](/api/auth/login). The `usertype` encoded in the token must be `"Police"`, `"Bank"`, or `"Court"`.
</ParamField>

### Body parameters

<ParamField body="applicationNo" type="number" required>
  The unique 4-digit identifier of the complaint to verify. Obtain this from the [list unverified complaints](/api/complaints/download#get-formuploaddownloadformsnotverify) endpoint.
</ParamField>

<ParamField body="verifybypolice" type="string" required>
  The verification decision. Use `"yes"` to approve the complaint.

  Despite the field name, this value is used by all roles — police, bank, and court — to record their decision. The server routes it to the correct database field based on the caller's token.
</ParamField>

## How it works

The server inspects the `usertype` claim in the provided token and updates the corresponding fields on the matching complaint document:

| Role | Fields updated |
|---|---|
| `Police` | `verifybypolice`, `verifybypoliceemail` |
| `Bank` | `verifybybank`, `verifybybankemail` |
| `Court` | `verifybycourt`, `verifybycourtemail` |

The email address of the verifying officer is automatically read from the token and stored alongside the decision.

<Warning>
  If the `usertype` in the token is not `"Police"`, `"Bank"`, or `"Court"`, the server returns `false` without making any changes. Victim accounts cannot call this endpoint.
</Warning>

## Verification flow

Complaints move through a sequential verification pipeline:

1. **Police** review newly submitted complaints and record a decision.
2. **Bank** officers review police-verified complaints that involve a bank transaction (`bankbool: true`).
3. **Court** officers review police-verified complaints and issue a court-level determination.

See [List and retrieve complaints](/api/complaints/download) for how to query complaints pending verification at each stage.

## Response

All responses return HTTP `200` with a JSON body.

<ResponseField name="(root)" type="boolean">
  `true` when the complaint was found and the verification record was saved successfully. `false` when the complaint does not exist, the role is not authorised, or a server error occurs.
</ResponseField>

## Examples

<CodeGroup>

```bash cURL — police verification
curl --request POST \
  --url https://your-api-host/formupload/verifybypolice \
  --header 'Content-Type: application/json' \
  --header 'token: <police-jwt-token>' \
  --data '{
    "applicationNo": 4821,
    "verifybypolice": "yes"
  }'
```

```javascript JavaScript — police verification
const response = await fetch('https://your-api-host/formupload/verifybypolice', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    token: '<police-jwt-token>',
  },
  body: JSON.stringify({
    applicationNo: 4821,
    verifybypolice: 'yes',
  }),
});

const success = await response.json();
console.log(success); // true
```

</CodeGroup>
